# 8x04 - Bug Bounty Guide

如何将研究成果转化为价值。

本章侧重“工程化交付”：如何把漏洞从研究阶段整理成可被厂商高效处理的报告与证据包。

## 1. 知名项目
- **ASRP (Android Security Rewards Program)**: Google 官方项目。
- **ZDI (Zero Day Initiative)**: 趋势科技旗下的漏洞收购平台。
- **厂商私有项目**: 三星、华为、小米等。

补充：不同项目对“范围、影响、复现质量”的要求差异很大，提交前需先确认：

- 资产/版本是否在范围内
- 是否允许 root/工程机前置条件
- 是否要求提供补丁建议或根因分析

## 2. 报告撰写
- **PoC (Proof of Concept)**: 必须是可复现的。
- **影响分析**: 明确漏洞的危害（如 RCE, LPE）。
- **修复建议**: 帮助厂商快速解决问题。

### 2.1 报告结构模板

- 摘要：一句话说明漏洞类型与影响
- 影响范围：受影响版本/设备/组件
- 复现步骤：最小步骤 + 所需环境
- 证据：日志、堆栈、截图/录屏（按需）
- 根因分析：关键代码路径与缺失校验
- 修复建议：最小修复点与回归风险

### 2.2 复现质量要点

- 固定环境（版本号、patch level、配置开关）
- 输入最小化（最小 Intent/最小样本/最小数据包）
- 给出失败/成功的判定标准（例如出现特定日志/崩溃）

### 2.3 影响分析要点

- 触发条件：是否需要用户交互、是否需要本地权限
- 可达性：入口是否可由第三方应用或外部输入触发
- 缓解机制：SELinux/seccomp/进程拆分是否限制影响面

## 3. 时间线与合规

常见实践：

- 与厂商沟通修复窗口
- 在修复发布前避免公开可复现细节
- 保留复现证据与版本信息用于后续确认

## 参考（AOSP）
- https://source.android.com/docs/setup/contribute/report-bugs — AOSP 官方问题提交流程（怎样写复现步骤/日志最有效）。
- https://source.android.com/docs/security/bulletin — 安全公告入口：用于对齐披露窗口、补丁发布节奏与版本影响范围。
